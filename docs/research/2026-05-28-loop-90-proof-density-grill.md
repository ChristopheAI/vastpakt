# Loop 90 - Proof Density Grill

## Question

Does the remaining proof-density uncertainty mean the homepage needs more proof, different proof, or weaker proof language?

## Sources used

- Current homepage examples after loop 89.
- Local practice-case notes for:
  - three-mailbox extraction;
  - Forthing-style WordPress/plugin case;
  - workshop inventory app;
  - BAVAST Word report case.
- Local workdirectory spot checks:
  - Triscalis export files and extraction script;
  - Forthing plugin ZIPs, validation scripts and plugin entry files;
  - ML Coating inventory app files and UI labels.
- Local Swipefile proof-system spec.
- Exa competitor case-study pass:
  - We Are / Symons;
  - iterates / Homeprespective;
  - iterates / SO+;
  - Zodi / Van Cauwenbergh Transport;
  - related custom-software case pages.

## Finding

Competitor case pages usually build credibility with some mix of:

- named client;
- before state;
- built object;
- feature list;
- result or metric;
- testimonial.

Vastpakt should not copy that case-study tone. The current homepage has enough local proof objects for a compact homepage grid, but not enough externally published proof to add heavier claims, testimonials, logos or broad result language.

The weak spot was smaller:

`Vastpakt maakte zichtbaar`

appeared as the repeated label inside all four cards. It is true, but it makes the case cards slightly self-referential. Swipefile discipline says the object should carry proof where possible.

## Decision

Change the repeated case label from:

`Vastpakt maakte zichtbaar`

to:

`Wat zichtbaar werd`

## Why

This keeps the same factual content but shifts attention from Vastpakt's claim to the work object:

- voorraad webapp;
- WordPress/plugin parts;
- mailbox workbook;
- BAVAST report steps.

## Boundary

Do not add logos, testimonials, public client names, stronger metrics, social-proof language or broad result claims until there is explicit permission and hard proof.
